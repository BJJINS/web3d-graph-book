# 片元着色器（Fragment Shader）

片元着色器运行在[光栅化之后](../WebGPU/graphicsRendering.md#rasterization)，会对**每一个片元**执行一次。  
它最核心的职责是：**决定这个片元最终要输出什么颜色，必要时也可以改写深度。**

如果把[顶点着色器](./vertexShader.md)理解成“先把形状摆到正确位置”，那么片元着色器更像是“给这些片元决定外观”。

## 先看一个最小例子

```wgsl
@fragment
fn fsMain() -> @location(0) vec4f {
    return vec4f(1.0, 0.647, 0.0, 1.0);
}
```

这段代码已经包含了片元着色器最常见的 3 个关键信息：

1. `@fragment` 表示这是片元阶段入口
2. `@location(0)` 表示把结果写到第 0 个颜色输出位置
3. 返回值是 `vec4f`，通常按 `(r, g, b, a)` 理解

这里返回的是一组固定颜色：

- 红色：`1.0`
- 绿色：`0.647`
- 蓝色：`0.0`
- Alpha：`1.0`

因此，这个着色器会把当前片元输出成橙色。

## 片元到底是什么

很多初学者会把“片元”和“像素”直接当成一回事，但更准确的理解是：

- **像素**是最终显示在屏幕上的结果
- **片元**是渲染过程中产生的候选结果

为什么要分开？

因为一个屏幕位置上，可能先后有多个图元都生成片元。  
后面的深度测试、模板测试、混合等步骤会决定：

- 哪些片元能留下来
- 哪些片元会被丢弃
- 颜色该怎样和已有结果组合

所以片元着色器面对的是“还在渲染流程中的候选像素”，而不是已经确定显示到屏幕上的最终像素。

## 片元着色器的输入从哪里来

片元着色器最常见的输入，不是直接来自顶点缓冲，而是来自两个地方：

- 光栅化阶段提供的片元信息
- 顶点着色器传下来的跨阶段数据

其中第二种最重要，也最常见。  
也就是说，顶点着色器先把某些值写出去，GPU 再在图元内部对这些值做插值，最后片元着色器读取插值后的结果。

这条“顶点阶段输出 → 片元阶段输入”的数据通道，就是[Inter-stage](../WebGPU/inter-stage.md)。

## 最常见的输入方式：接收顶点阶段传下来的数据

例如，每个顶点自带一个颜色。顶点着色器把颜色输出后，片元着色器就可以把这个颜色读进来：

```wgsl
struct VertexOut {
    @builtin(position) clipPos: vec4f,
    @location(0) color: vec3f,
};

@fragment
fn fsMain(inData: VertexOut) -> @location(0) vec4f {
    return vec4f(inData.color, 1.0);
}
```

这里最值得注意的是 `color` 这个字段：

- 顶点阶段把它写到 `@location(0)`
- 片元阶段通过同样的 `@location(0)` 接收到它

如果三角形的三个顶点颜色不同，那么三角形内部并不会只出现三种离散颜色，而会出现**连续过渡**。  
这正是因为 GPU 会在图元内部对这些逐顶点数据做插值。

::: tip 一个很重要的直觉
片元着色器通常读到的，不是“某个顶点原封不动的值”，而是**插值之后的结果**。
:::

## 片元着色器里常见的内建值

除了接收跨阶段数据，片元着色器还可以访问一些内建值。

| 内建值 | 类型 | 读写 | 作用 |
| --- | --- | --- | --- |
| `@builtin(position)` | `vec4f` | 读 | 当前片元的位置与相关屏幕空间信息 |
| `@builtin(frag_depth)` | `f32` | 写 | 当前片元输出深度 |
| `@builtin(front_facing)` | `bool` | 读 | 当前片元是否来自正面图元 |
| `@builtin(sample_index)` | `u32` | 读 | 当前采样编号 |
| `@builtin(sample_mask)` | `u32` | 读 / 写 | 当前采样覆盖掩码 |

对入门阶段来说，最值得先认识的是前 3 个。

### `@builtin(position)`

它表示当前片元的位置相关信息。  
你可以先把它理解成：**这个片元现在位于屏幕空间的什么位置，以及它当前带着什么深度信息。**

在片元阶段里，它是输入值，不是输出值。

### `@builtin(frag_depth)`

如果你想显式写出片元深度，可以使用它。  
但在大多数入门示例中，片元着色器通常只输出颜色，不会手动改深度。

### `@builtin(front_facing)`

当你想区分“正面”和“背面”片元时，它很有用。

```wgsl
@fragment
fn fsMain(@builtin(front_facing) isFront: bool) -> @location(0) vec4f {
    if (isFront) {
        return vec4f(0.2, 0.8, 0.3, 1.0);
    }
    return vec4f(0.9, 0.3, 0.3, 1.0);
}
```

这段代码的意思是：

- 来自正面图元的片元输出绿色
- 来自背面图元的片元输出红色

## 片元着色器最常见的输出：颜色

在入门阶段，片元着色器最常做的事情就是返回颜色：

```wgsl
@fragment
fn fsMain() -> @location(0) vec4f {
    return vec4f(1.0, 0.4, 0.2, 1.0);
}
```

这里的 `@location(0)` 不是随便写的。它需要和 JavaScript 侧 render pipeline 中的颜色目标对应起来。

例如：

```js
const pipeline = device.createRenderPipeline({
    layout: "auto",
    vertex: {
        module: shaderModule,
        entryPoint: "vsMain",
    },
    fragment: {
        module: shaderModule,
        entryPoint: "fsMain",
        targets: [{ format: canvasFormat }],
    },
});
```

这段配置里：

- `entryPoint: "fsMain"` 指定片元着色器入口函数
- `targets[0]` 表示第 0 个颜色目标

因此，WGSL 里的 `@location(0)`，就对应这里的第一个颜色输出目标。

## 为什么片元着色器经常和顶点着色器一起看

单独看片元着色器时，很容易误以为它只是“返回一个颜色”。  
但在真实渲染里，它通常和顶点着色器紧密配合：

1. 顶点着色器决定图元形状与逐顶点输出
2. GPU 对图元内部数据做插值
3. 片元着色器读取插值后的结果并输出颜色

所以很多时候，片元着色器的重点不只是“怎么算颜色”，还包括：

- 它接收了哪些跨阶段输入
- 这些输入的 `@location(...)` 是否匹配
- 它的输出要写到哪个颜色目标

如果你对这部分还不够熟悉，建议配合阅读：

- [顶点着色器](./vertexShader.md)
- [跨阶段数据传递（Inter-stage）](../WebGPU/inter-stage.md)

## 什么是插值

看下面这组逻辑：

- 顶点 A 是红色
- 顶点 B 是绿色
- 顶点 C 是蓝色

顶点着色器把这 3 个颜色输出之后，GPU 会在三角形内部做插值。  
于是离 A 更近的片元会更偏红，离 B 更近的片元会更偏绿，离 C 更近的片元会更偏蓝。

所以片元着色器经常不是在“凭空发明颜色”，而是在：

- 读取插值后的颜色
- 再决定最终怎样输出

这也是为什么很多三角形示例里，片元阶段代码明明很短，画面却能出现明显的渐变效果。

你可以使用这个[示例](../WebGPU/graphicsRendering.md#triangleGridCanvas-demo)来体验颜色插值。

## 初学者最容易犯的错

### 1. 把片元和像素完全等同

片元是渲染过程中的候选结果，后面还可能经过测试、丢弃或混合。  
直接把它理解成“最终像素”会让很多渲染现象变得难以解释。

### 2. 片元输入和顶点输出的 `@location(...)` 对不上

例如顶点阶段写的是：

```wgsl
@location(1) color: vec3f
```

片元阶段却读成：

```wgsl
@location(0) color: vec3f
```

这样就接不上。

### 3. 片元输入类型和顶点输出类型不一致

不仅 `location` 要一致，类型也要一致。  
比如顶点阶段输出 `vec3f`，片元阶段就不应该按 `vec2f` 去接。

### 4. 忘了颜色输出和 pipeline 的目标要对应

WGSL 里写了 `@location(0)`，JavaScript 侧就应该有能接收第 0 个颜色输出的 `targets[0]` 配置。

### 5. 误以为片元着色器直接读取顶点缓冲

更常见的路径是：

```text
顶点缓冲 → 顶点着色器 → 插值 → 片元着色器
```

也就是说，片元阶段读到的通常是“传下来的结果”，而不是直接重新去取顶点输入。

## 可以这样记住片元着色器

学片元着色器时，可以先抓住下面这条主线：

1. 图元经过光栅化后产生片元
2. 片元着色器对每个片元执行一次
3. 它读取当前片元的输入信息
4. 它最常见的任务是输出颜色，必要时也可以处理深度
5. 输出结果后，后面还可能继续经过深度、模板和混合等步骤

这样一来，片元着色器就不再只是“一个返回 `vec4f` 的函数”，而是整条渲染链路中真正负责“决定表面外观”的阶段。
