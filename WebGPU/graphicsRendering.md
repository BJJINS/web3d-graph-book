# 图形渲染流程

这一章只讲一件事：WebGPU 是如何把“顶点数据”一步步变成“屏幕像素”的。这里不放任何代码，我们介绍渲染这条流水线的概念，后续章节再把每一步落到具体 API 与案例上。

如果你是初次接触图形渲染，在看完本章后可能会感到困惑，但是没关系，下一章是一个极简的案例，我会在[绘制一个三角形](./triangleExample.md)的案例中将代码与本章节的流程关联。

## 你在做的事情是什么

一次最典型的渲染可以被概括为：

- 在 CPU 侧准备数据与渲染状态（缓冲、纹理、管线、绑定）
- 录制一段 GPU 命令（开始渲染通道、设置状态、发起绘制、结束通道）
- 让 GPU 执行命令，把结果写到一张“颜色目标纹理”
- 把这张纹理呈现到屏幕（Canvas）

从学习角度看，你可以把 WebGPU 当成“录制命令 + 提交命令”的系统：你不是每画一个像素就调用一次 API，而是描述一段工作流程，交给 GPU 批量执行。

## 流程拆解：每一步在做什么

### 1. 输入数据（Input Data）

输入数据通常由三部分构成：

- 几何数据：顶点缓冲与索引缓冲（位置、颜色、法线、纹理坐标等属性）
- 常量数据：相机矩阵、时间、材质参数等（常量缓冲 / Uniform）
- 纹理数据：贴图与采样器（决定如何从纹理取样）

这些数据本质上都是“GPU 可读的内存资源”。渲染前的很多工作，就是把 CPU 内存里的数据组织好，并传到 GPU 能访问的地方。

::: tip
下一章节中我们要绘制的三角形案例的输入数据只有**顶点缓冲**，一个三角形由三个点组成，我们把这三个点叫作**顶点**，然后我们将这三个顶点放到 GPU 缓冲区中，方便 GPU 在渲染时获取数据。
:::

### 2. 顶点着色器（Vertex Shader）

顶点着色器是几何处理的入口。它对“每个顶点”执行一次，绘制一个三角形时顶点着色器会执行 3 次。

顶点着色器用来完成两类任务：

- 计算顶点的位置：把模型空间的点变换到裁剪空间，为后续裁剪与光栅化做准备
- 输出顶点属性：把颜色、法线、纹理坐标等写到输出里，供后续插值给片元使用

你可以把它理解为：把“离散的顶点信息”变成“图元可以理解的顶点描述”。

::: tip
现在你可以这样理解顶点着色器的作用：
为了绘制一个三角形，上一步中我们输入了三个顶点数据。这是如果我们如果想要平移、旋转、缩放三角形，我们可以在顶点着色器中进行这些操作，最后你可以将这些处理完的顶点数据`return`出去。
这些顶点会进入下一步：图元装配。
:::

### 3. 图元装配（Primitive Assembly）

这个阶段主要由两个任务：

1. 图元组装
2. 图元处理

WebGPU 只可以绘制点、线段和三角形，默认情况下使用三个顶点绘制一个三角形。

**图元组装**指顶点数据根据设置的绘制方式被结合成完整的图元。
列如，在点的绘制方式下每个图元仅需要一个单独的顶点，在此方式下每个顶点为一个图元；
在线段绘制方式下每个图元则需要两个顶点；
在三角形绘制方式下需要三个顶点构成一个图元。

::: tip
WebGPU 只可以绘制点、线段和三角形，默认情况下使用三个顶点绘制一个三角形，后续的章节我们会介绍如何切换绘制方式。
:::

**图元处理**最重要的工作是剪裁，其任务是图元在剪裁空间之外的部分。
对于点绘制方式来说，点剪裁就是简单的接收、拒绝顶点。
线段或者多边形的剪裁可能需要额外增加顶点，这具体取决于直线或者多边形与剪裁空间的位置关系。

随着观察角度和位置的改变，不能总能看到特定3D物体上某个图元的全部。比如：一个三角形的“部分”在屏幕之外，这个“部分”需要被剪掉。这时渲染在屏幕上的就不是一个三角形而是一个多边形。

![剪裁空间中的图元剪裁示意图](/clip-space-clipping.svg)

### 4. 光栅化（Rasterization）

虽然虚拟3D世界的几何信息是三维的，但由于目前用于显示的设备都是二维的，所以在真正执行光栅化工作之前，需要将虚拟3D世界中的物体投影到视平面（可以理解为屏幕）上。

需要注意的是由于观察的角度和位置不同相同的3D物体投影到视平面上的可能产生不同的效果：

![从 3D 到 2D 的投影示意图](/view-projection-demo.svg)

另外，由于在虚拟3D世界中物体的几何信息一般采用连续的量来表示，因此投影的平面结果也是用连续量来表示的。
但目前显示设备屏幕都是离散化的(由一个个的像素组成)因此还需要将投影结果离散化，将其分解为一个个离散化的小单元，这些小单元一般称为**片元**。

其实每个片元都对应于帧缓冲中的一个像素，之所以不直接称为像素是因为3D空间的物体是可以相互遮挡的。一个3D场景最终显示到屏幕上虽然是一个整体，但每个3D的每个图元都是独立处理的。这就可能出现以下情况，系统先处理的是位于离观察点较远图元，其光栅化成为一组片元，暂时送入帧缓冲的对应位置。

但在后面继续处理离观察点近的图元时也光栅化出了一组片元，两组片元又对应到帧缓冲中同一个位置，这时距离近片元将覆盖距离远的片元(如何检测覆盖是在**深度检测**阶段完成的)。因此某个片元就不一能成为最终屏幕上的像素，这样称为像素就不准确了，可以将其理解为候选像素。

每个片元包含对应的顶点坐标、顶点颜色、顶点纹理坐标以及顶点深度等信息，这些信息是系统根据投影前此片元对应3D空间中的位置及与此片元相关的图元中各顶点信息进行**插值计算**而生成的。

你可能还是不明白什么是插值计算，这里我举个例子：使用 WebGPU 绘制一个三角形，我们的输入数据是三个顶点和三个颜色，每个顶点对应一个颜色，并且三个颜色互不相同，那么三角形内部的某个片元的颜色是什么呢？

这需要根据三个顶点的颜色来确定，这个过程就叫**插值计算**。

![顶点颜色插值示意图](/attribute-interpolation.svg)

### 5. 片元着色器（Fragment Shader）

片元着色器对“每个片元”执行一次，用来计算最终写入颜色目标的值，例如：

- 计算颜色（基于插值后的颜色/法线/纹理坐标等）
- 采样纹理并组合材质参数
- 决定是否丢弃片元（让它不写入任何结果）

如果你把顶点着色器理解为“处理形状”，片元着色器更像是“决定外观”。

### 6. 深度/模板测试与混合（Depth/Stencil & Blending）

片元着色器输出之后，并不一定立刻写进帧缓冲，通常还会经过一组固定功能测试与混合步骤：

- 深度测试：决定哪个片元在前面，解决遮挡关系
- 模板测试：用一个额外缓冲控制哪些区域允许/禁止绘制
- 颜色混合：把“新颜色”和“已有颜色”按规则混合（例如 Alpha 混合实现半透明）

这些步骤决定了“写入的条件”和“写入的方式”，是渲染正确性与表现力的重要来源。

### 7. 输出到目标（Render Target / Pixels）

通过以上步骤，GPU 最终把结果写到一个颜色目标（通常是一张纹理）。如果这个目标与 Canvas 的交换链关联，浏览器就可以把它呈现到屏幕上，成为你看到的像素结果。

## 你需要掌握的三层模型

为了在后续章节更顺畅地阅读 API，你可以把 WebGPU 渲染理解为三层：

- 资源层：缓冲、纹理、采样器，以及它们承载的数据
- 管线层：渲染管线与绑定布局，决定“数据如何被解释、着色器如何运行”
- 命令层：渲染通道与绘制命令，决定“这一帧具体做哪些事情、按什么顺序做”

当画面出现问题时，也可以用这三层快速定位：

- 形状不对：常见在资源层（顶点数据/索引）或顶点阶段（变换）
- 外观不对：常见在片元阶段（材质/纹理采样）或混合/深度配置
- 什么都没画出来：常见在命令录制顺序、目标格式不匹配、或测试/裁剪把片元都过滤掉

## 从“像素网格”建立直觉

学习渲染流程时，一个很有帮助的直觉是：屏幕是一张像素网格，三角形在这张网格上“覆盖”并生成片元，然后每个片元计算颜色并写入。后续的示例章节会用一个非常小的交互演示，帮助你把“光栅化覆盖 + 插值 + 片元着色”这件事在脑子里彻底具体化。
