WebGPU 是新一代 Web 图形与计算 API。它让浏览器可以更直接、更高效地使用现代 GPU，既能在 Canvas 上做图形渲染，也能把通用计算任务交给 GPU 并行执行。

## WebGPU 从哪里来

在 WebGPU 之前，浏览器里最常用的 GPU 图形接口是 WebGL。WebGL 的设计理念更接近早期的 OpenGL ES：状态机式的接口易于上手，但在现代 GPU 的工作方式、性能可控性、以及“图形 + 计算”的统一能力上，都有明显的历史包袱。

随着桌面与移动端 GPU 逐步转向更“显式”的底层 API（如 Direct3D 12 / Vulkan / Metal），Web 平台也需要一个能够匹配现代 GPU 特性的标准化接口。WebGPU 正是在这样的背景下诞生的：它由 W3C 的 “GPU for the Web” 社区组推动，目标是为 Web 提供一个面向现代 GPU 的、跨平台的图形与计算标准。

你可以把 WebGPU 理解成：

- 在浏览器里可用的现代 GPU 抽象层
- 以更接近底层 API 的方式组织资源与提交命令
- 同时覆盖渲染（Render）与计算（Compute）

## WebGPU 开发流程

既然你已经了解了 WebGPU 是什么，我想简单介绍一下 WebGPU 开发。一般来说，构成 WebGPU 应用程序的代码大约 90% 是 JavaScript，10% 是 WebGPU 着色器语言（[WGSL](../WGSL/introduction.md)）。具体来说，编写一个有一定复杂度的 WebGPU 应用程序需要三个步骤：

1. 创建访问GPU所需的JavaScript对象。
2. 创建包含需由GPU处理的数据和指令的JavaScript对象。
3. 将数据和指令发送到GPU执行。

步骤1和步骤2依赖于 JavaScript，如果你已经熟悉这门语言，那么大多数 API 都不会带来任何困难——你只需创建对象并调用它们的方法即可。但需要创建的对象很多，要对它们进行跟踪可能会有难度。值得庆幸的是，一旦你编写出一个可运行的应用程序，在其他应用程序中使用类似的代码就会变得简单直接。

第三步是困难的部分。要在 GPU 上执行代码，你需要使用 WebGPU 着色器语言（WGSL）编写特殊程序（着色器）。WGSL 类似于 C 编程语言，但它经过简化，专注于数字运算。也就是说，你不需要字符串处理函数，而且可能也不会大量使用指针。其目标是高速处理数字，你对线性代数的理解越透彻，编写着色器就会越轻松。

## WebGPU 能做什么

WebGPU 的能力可以粗略分为两类：

1. 图形渲染，在HTML画布中显示图形
2. GPU 计算，计算应用程序执行计算任务，例如对矩阵进行运算或使用机器学习分析数据

## 图形渲染（Graphics Rendering）

图形渲染的目标是把三维/二维几何与材质信息，变成屏幕上的像素。WebGPU 通过渲染管线（Render Pipeline）把“输入数据 → 着色器 → 光栅化 → 输出目标”串成一条稳定的流水线，并提供缓冲区、纹理、采样器等资源类型，让你把数据高效地送进 GPU。

## 在 GPU 上执行计算（GPU Computing）

GPU 不仅能画图，还非常擅长对大量数据做同一类运算。WebGPU 提供计算管线（Compute Pipeline），让你用计算着色器把图像处理、粒子模拟、物理/几何预处理、并行归约等任务放到 GPU 上执行，并把结果写回缓冲区或纹理，继续用于下一步渲染或回读到 CPU。

计算类应用通常比图形类应用更容易编写代码。需要创建的JavaScript对象更少，也无需访问画布。应用程序不必关注顶点缓冲区，而是可以将数据存储在存储缓冲区中，后者使用起来更简便。

::: tip WebGPU 不是引擎
WebGPU 更像“工具箱”：它负责资源管理与命令提交，不会替你做相机、场景图、材质系统等高层封装。你可以在它之上构建自己的渲染框架，或使用已有引擎/库的 WebGPU 后端。
:::
